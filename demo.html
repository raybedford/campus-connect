Development Roadmap

5-Week Build Plan

From zero to a production-grade encrypted messaging platform in five focused sprints.

WEEK 01

Foundation & Infrastructure

  • Set up Supabase project (PostgreSQL, Auth, Realtime, Storage)
  • Design and deploy database schema — schools, profiles, conversations, messages, public_keys, push_tokens tables
  • Implement Row-Level Security (RLS) policies for all tables
  • Create database triggers for auto profile creation and school association
  • Scaffold React + TypeScript + Vite frontend
  • Configure Zustand state stores (auth, conversations, messages, presence)
  • Set up Vercel deployment pipeline
WEEK 02

Authentication & End-to-End Encryption

  • Implement Supabase Auth with .edu email validation
  • Build signup, login, and password reset flows
  • Generate NaCl Box keypairs (Curve25519 + XSalsa20-Poly1305) on signup
  • Store private keys in IndexedDB via idb-keyval — never leaves device
  • Publish public keys to Supabase public_keys table
  • Implement client-side message encryption and decryption
  • Add encrypted file upload/download to Supabase Storage
  • Multi-factor authentication support
WEEK 03

Real-Time Messaging & Core Features

  • Integrate Supabase Realtime for live message delivery (PostgreSQL Changes)
  • Build direct message (1:1) conversation flow
  • Build group chat system with role management (admin / member)
  • Implement typing indicators via Supabase Broadcast channels
  • Add online presence tracking
  • Read receipts with last_read_at timestamps
  • Message history with lazy loading
  • Encrypted file sharing (ephemeral uploads)
WEEK 04

UI/UX Polish & Discovery

  • Build school-wide student directory scoped by university
  • One-click DM from directory
  • Gold & Black theme system with CSS custom properties
  • Responsive mobile-first layout (Flexbox / Grid)
  • Custom SVG campus building icon
  • Conversation list with unread indicators
  • Message bubbles with sender info and timestamps
  • Settings page with profile management
WEEK 05

Testing, Security Audit & Launch

  • Full TypeScript type coverage and ESLint cleanup
  • Cross-school data isolation verification
  • E2EE key lifecycle testing (generation, backup, restore)
  • RLS policy audit — ensure no data leaks
  • Performance optimization and database indexing
  • Vercel production deployment with environment variables
  • Render fallback deployment configuration
  • 14-day delayed account deletion safety net
  • Documentation and CHANGELOG finalization

System Design

Architecture Principles

Zero-knowledge, serverless, and encrypted by default.

⚡ Serverless Architecture

No backend API server. All logic runs client-side or via Supabase row-level security policies. Zero infrastructure to manage.

🔐 Client-Side Encryption

Messages are encrypted before leaving the browser. The server stores ciphertext only — it never sees plaintext content.

🛡 Row-Level Security

PostgreSQL RLS policies enforce access control at the database level. No middleware, no API layer — the database is the firewall.

⚡ Real-Time WebSockets

Supabase Realtime delivers messages instantly via WebSocket connections. Typing indicators and presence use broadcast channels.

👁 Zero-Knowledge Design

Private keys are generated on-device and stored in IndexedDB. They never leave the user's browser — not even to the server.

🎓 .edu Verified

Only university email addresses are accepted. Schools are auto-detected from email domains, ensuring a trusted campus community.

Six tables, fully RLS-protected, with automatic triggers for profile and school creation.

schools

  • domain
  • name
  • logo_url
  • created_at
profiles
  • PK id
  • email
  • display_name
  • FK school_id
  • avatar_url
  • phone_number
  • preferred_language
  • is_verified
conversations
  • PK id
  • type
  • name
  • FK school_id
  • FK created_by
  • created_at
  • updated_at
conversation_members
  • FK conversation_id
  • FK user_id
  • role
  • joined_at
  • last_read_at
messages
  • PK id
  • FK conversation_id
  • FK sender_id
  • content E2EE
  • message_type
  • file_url
  • created_at
public_keys
  • FK user_id
  • public_key
  • created_at
